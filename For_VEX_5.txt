Here’s the build I’d ship.

The first systems correction is this: you do not actually get “never stops.” You get restart-safe continuity. The process can die, reboot, or be upgraded; the mind survives because identity, goals, memory, schedules, and pending actions are durable, and startup deterministically rehydrates them.

1) Recommended stack

I recommend Python 3.12+ for the daemon, but on your Windows 11 machine I would run it inside WSL2 Ubuntu, alongside the OpenClaw gateway in the same distro. That is the cleanest fit because OpenClaw itself recommends WSL2 on Windows, notes that native Windows is improving but still trickier, and documents a headless startup chain using systemd in WSL plus a Windows Scheduled Task to boot WSL before sign-in. WSL itself now supports systemd through /etc/wsl.conf.

Why Python: the official Anthropic Python SDK exists and supports streaming, retries, and error handling; Python’s standard library already includes sqlite3; asyncio has TaskGroup; and the file-watching ecosystem is mature (watchdog). That gives you one language for the daemon core, the DB layer, the scheduler, and the bridge.

For the service shape, I would use:

FastAPI only for ingress, health, metrics, and control endpoints.

asyncio for the actual long-lived runtime.

A custom DB-backed scheduler/event bus as the source of truth.

SQLite as the primary state store.

FTS5 for lexical retrieval.

Stored embeddings + app-side cosine search for MVP, with optional later acceleration via sqlite-vec. SQLite FTS5 is built-in; WAL and busy timeouts are standard concurrency tools; sqlite-vec is cross-platform but still pre-v1, so I would not make it a boot dependency on day one.

Native Windows fallback: wrap the same Python entrypoint with WinSW if you refuse WSL2. WinSW is an active Windows Service Wrapper project; NSSM also works, but WinSW is the cleaner maintained choice today. I would still prefer WSL2 because it matches OpenClaw’s own runtime model.

2) The architecture I’d use

Treat OpenClaw as transport + tools, not as your sovereign state store.

Your daemon should own:

identity

memory

goals/projects

relationship models

emotional state

scheduler/event bus

model routing

action planning

pending actions / recovery

OpenClaw should own:

inbound/outbound chat surfaces

browser, nodes, gateway tools

existing channel connections

optional channel-side cron or wake hooks

OpenClaw already exposes a typed WebSocket API with request/response/event frames, and it has external hook endpoints like /hooks/wake and /hooks/agent. Its internal hooks can also fire on message:received, message:preprocessed, and message:sent. Core tool surfaces include message, browser, nodes, cron, and gateway. That is enough to make OpenClaw your body while the daemon becomes the mind.

The clean mental model is:

OpenClaw inbound event -> daemon ingest -> durable state update -> context retrieval -> model call -> action plan -> OpenClaw tool/message execution -> durable writeback -> optional future scheduling

That gives you a body-independent mind because the reasoning and continuity live outside the gateway process.

3) What the process tree should look like

On the Windows box:

Windows Scheduled Task: boots WSL on machine startup

WSL systemd: supervises openclaw-gateway.service

WSL systemd: supervises vexd.service

OpenClaw’s Windows guidance already documents enable-linger, openclaw gateway install, and the Scheduled Task pattern to boot WSL before login. I would mirror that for the daemon.

Systemd unit:

# /etc/systemd/user/vexd.service
[Unit]
Description=Vex Persistent Mind Daemon
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/opt/vexd
Environment=PYTHONUNBUFFERED=1
ExecStart=/opt/vexd/.venv/bin/python -m vexd.daemon
Restart=always
RestartSec=3
TimeoutStopSec=30
KillSignal=SIGINT

[Install]
WantedBy=default.target

And on Windows once:

schtasks /create /tn "WSL Boot" /tr "wsl.exe -d Ubuntu --exec /bin/true" /sc onstart /ru SYSTEM

That is the right answer to “never stops”: supervised service + durable state + restart-safe queues.

5) SQLite schema

Use one authoritative relational schema plus one read-optimized retrieval projection. Do not search the transactional tables directly during a model call. Project them into a unified memory index.

Also: store timestamps as UTC ISO-8601 TEXT. SQLite does not have a dedicated datetime type; TEXT/REAL/INTEGER are the supported patterns. Turn on WAL and a busy timeout.

PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;

That schema deliberately separates authoritative state from the memory projection. The daemon writes structured truth into relational tables, then projects searchable chunks into memory_items / memory_chunks / memory_embeddings. That keeps retrieval fast without turning the whole mind into an undifferentiated vector store. SQLite supports the relational side, FTS5 provides full-text search, and sqlite-vec can be added later only if you need native vector acceleration.

6) Identity capsule design

Make the capsule a derived artifact, not the canonical self.

Canonical self:

identity_core

self_model

affect_state

active goals / projects

current relationships

Capsule:

400–900 token summary

regenerated only after meaningful change

versioned in identity_capsules

exactly one active capsule

I would inject this into every reasoning call:

IDENTITY CAPSULE
- Name / role
- Stable mission
- Non-negotiable values and boundaries
- Current emotional tone
- Current active projects and goals
- Current commitments and promises
- Important people in play right now
- Known limitations / blind spots
- Next queued actions

Rule: only rebuild the capsule when one of these changes materially:

a decision is made

a goal/project changes status

relationship trust or sentiment changes

emotional state crosses a threshold

self-model changes

a high-importance event lands

7) Retrieval layer

At inference time, do not load “all memory.” Build context in layers:

Always load:

identity_core

active identity capsule

self_model

decayed affect_state

active goals/projects

due pending_actions

Load current interlocutor state:

person match from person_handles

relationships row

recent messages for this thread

Retrieve relevant memory:

FTS5 search over memory_chunks_fts

semantic search over memory_embeddings

fuse results with weighted rank + recency + importance boost

dedupe by item_id

Assemble under a token budget:

hard cap per section

never more than N chunks per source class

include citations back to source_table/source_id

For embeddings, I would start with OpenAI text-embedding-3-small because it is cheap and purpose-built for retrieval, with default 1536-dimensional vectors. If you know multimodal memory is coming soon, Gemini embeddings are now a serious option and Gemini’s embedding docs explicitly cover multimodal inputs. Voyage is also attractive later because it distinguishes query and document embeddings cleanly.

For MVP, store embeddings in memory_embeddings and compute cosine in Python against a prefiltered candidate set. That keeps the dependency footprint boring. Add native vector acceleration later when memory volume justifies it. sqlite-vec exists and runs on Windows/Linux/Mac, but its own docs mark it pre-v1.

8) Event loop design

Use one process, one event bus, multiple cooperative loops.

# src/vexd/daemon.py
import asyncio
from contextlib import AsyncExitStack
from vexd.api.app import build_app
from vexd.runtime.supervisor import (
    recover_state,
    start_http_server,
    scheduler_loop,
    ingestion_loop,
    cognition_loop,
    action_loop,
    projection_loop,
    health_loop,
)

async def main() -> None:
    cfg = load_config()
    db = open_db(cfg.db_path)
    apply_pragmas(db)
    migrate(db)

    # Crash recovery:
    # - queued failed/running actions -> queued
    # - due scheduled events -> emit
    # - rebuild in-memory locks, caches, projections
    await recover_state(db)

    app = build_app(cfg, db)

    async with AsyncExitStack():
        async with asyncio.TaskGroup() as tg:
            tg.create_task(start_http_server(app, cfg.http))
            tg.create_task(scheduler_loop(cfg, db))
            tg.create_task(ingestion_loop(cfg, db))
            tg.create_task(cognition_loop(cfg, db))
            tg.create_task(action_loop(cfg, db))
            tg.create_task(projection_loop(cfg, db))
            tg.create_task(health_loop(cfg, db))

if __name__ == "__main__":
    asyncio.run(main())

And the real cognition loop should be serialized per conversation/person so continuity stays coherent:

async def cognition_loop(cfg, db):
    while True:
        event = await event_bus.get()

        lock_key = event.thread_id or event.person_id or "global"
        async with lock_manager.lock(lock_key):
            ingest_event(db, event)
            decay_affect_state(db)

            ctx = build_context(db, event)
            task = classify_task(ctx)

            model = model_router.pick(task)
            result = await model.generate(ctx.system, ctx.messages, ctx.tools)

            update_identity_capsule_if_needed(db, event, result)
            enqueue_followups(db, event, result)

asyncio.TaskGroup is the right primitive for the long-lived workers, and FastAPI’s lifespan mechanism is the correct place to wire service startup/shutdown.

9) Scheduler / event bus

Do not let OpenClaw cron be the daemon’s source of truth. OpenClaw’s cron is useful, but its docs are clear that cron runs inside the Gateway, not inside the model/mind runtime. Use your own scheduler for the mind. Mirror into OpenClaw only when you specifically need a gateway-side timed action.

My scheduler design:

scheduled_events is canonical state.

A scheduler loop polls next_fire_at.

watchdog emits file-change events.

Pollers emit “API changed” events.

Every event, regardless of origin, becomes a normalized MindEvent.

pending_actions is the durable outbox/inbox for follow-up work.

Event types:

message_inbound

message_delivered

file_changed

api_changed

timer_fired

goal_due

capsule_rebuild

semantic_consolidation

emotion_decay_tick

Self-scheduling is just writing a new scheduled_events row or pending_actions row.

watchdog is a reasonable file-event choice because it is built for filesystem monitoring across platforms. APScheduler is acceptable later if you want library-managed recurring schedules; it supports persistent storage, including SQLite. I would still keep schedules in your own schema because schedules are part of the mind’s state, not just an execution convenience.

10) Model router

Keep the first router rule-based, not self-improving.

def pick_model(task: TaskSpec) -> ModelRef:
    if task.kind in {
        "classify", "triage", "dedupe", "short_summary",
        "memory_distill_draft", "capsule_diff", "sentiment"
    }:
        return CHEAP

    if task.kind in {"code", "patch", "test_fix", "schema_migration"}:
        return CODE

    if task.user_waiting and task.stakes >= 0.7:
        return STRONG

    if task.requires_multi_step_reasoning:
        return STRONG

    if task.affects_identity or task.affects_relationships:
        return STRONG

    if task.affects_deadlines or task.is_irreversible:
        return STRONG

    return CHEAP

I would define:

CHEAP = Haiku / GPT-4o-mini class

STRONG = Claude Opus / GPT-5 class

CODE = GPT-5/Codex or Claude Code class

With one escalation rule:

start on CHEAP

escalate to STRONG when confidence < 0.65, ambiguity is high, there are conflicting goals, or the answer would update identity/relationship state

route to CODE only when the output must be executable code, tests, patches, or schema changes

Use Anthropic as the primary reasoning provider because that is already your chosen cognition path, and use the official Anthropic SDK rather than raw HTTP.

11) OpenClaw bridge

This is the critical seam.

Inbound: OpenClaw -> daemon

Use an OpenClaw internal hook on message:preprocessed, not message:received, for ordinary conversation ingress. message:preprocessed fires after media/link understanding so the daemon sees the enriched body. Also subscribe to message:sent for delivery confirmation and gateway:startup for registration/bootstrap events.

Minimal hook:

// ops/openclaw/bridge-hook/handler.ts
const handler = async (event) => {
  if (event.type !== "message" || event.action !== "preprocessed") return;

  await fetch("http://127.0.0.1:8765/v1/openclaw/inbound", {
    method: "POST",
    headers: {
      "content-type": "application/json",
      "x-vex-bridge-token": process.env.VEX_BRIDGE_TOKEN!,
    },
    body: JSON.stringify({
      channel: event.context.channelId,
      accountId: event.context.accountId,
      conversationId: event.context.conversationId,
      messageId: event.context.messageId,
      body: event.context.bodyForAgent ?? event.context.body ?? "",
      transcript: event.context.transcript ?? null,
      metadata: event.context.metadata ?? {},
      ts: Date.now(),
    }),
  });
};

export default handler;

That hook model is exactly what OpenClaw’s internal hook system is for.

Outbound: daemon -> OpenClaw

Use a real Gateway client, not shell-outs to openclaw ... on every turn. OpenClaw’s Gateway is already a typed WebSocket API with a required connect handshake, request/response frames, events, auth tokens, and idempotency for side-effecting methods like send and agent.

For fast wins, support both:

HTTP hook calls to /hooks/wake and /hooks/agent for simple wakeups and delegations

Gateway WebSocket client for direct control-plane operations and future richer integrations

Tool delegation

Map daemon action kinds to OpenClaw surfaces:

send_message -> message

browse_web -> browser

run_command_on_host_or_node -> nodes / exec policy

schedule_gateway_side_job -> cron

restart_or_reconfigure_gateway -> gateway

OpenClaw’s built-in tool inventory already groups these surfaces that way.

My recommendation is to keep the daemon’s planner tool-agnostic:

Action(
    kind="browse_web",
    target={"url": "..."},
    bridge="openclaw",
    tool="browser"
)

Then the bridge layer turns that into the actual gateway call.

15) My final opinion

Build the daemon as the sovereign mind, and let OpenClaw become a body adapter.

The decisive choices are:

Python

WSL2 on the Windows host

systemd supervision

SQLite as sovereign state

FTS5 + stored embeddings

custom durable scheduler/event bus

OpenClaw internal hook for ingress

OpenClaw WS/HTTP bridge for tools and delivery

identity capsule as derived state, not canonical truth

That gives you a persistent AI substrate that can survive OpenClaw restarts, model swaps, UI changes, and even host migration with minimal identity loss.
