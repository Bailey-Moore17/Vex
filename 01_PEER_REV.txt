I read all the uploaded docs and both plans carefully. Here's where I land.

The peer review is strong. It caught the right things — transcript tailing as primary ingress is fragile, dual schedulers will split-brain, compaction on the critical path early is dangerous, and Phase A oversells what a database alone buys you without a read-path back into the turn loop. All correct.
The "sovereign vexd" rewrite is philosophically cleaner but practically brutal. Here's why that matters:
The sovereignty trap. If vexd owns model calls directly, you're rebuilding OpenClaw's entire agent loop from scratch — tool execution, streaming, channel message routing, browser automation, node commands, context window management, rate limiting. That's not a weeks-long project, that's months of full-time work for one person. The OpenClaw runtime already handles all of that. Ripping it out to prove Vex is "sovereign" is like building your own engine because you don't want to depend on the car. You still need the car.
The real insight buried in the sovereignty plan is not about who calls the API. It's about who owns truth. And you can get that without rebuilding the runtime. vexd owns state, memory, identity, and scheduling. OpenClaw owns execution — model calls, tools, channels, browser, nodes. The context-engine plugin is the membrane between them, not the brain. It reads from vexd's SQLite, assembles the capsule and relevant memory, hands it to OpenClaw's normal turn loop, and after the turn, writes deltas back. Vex's mind lives in vexd. OpenClaw is the body. That was actually the original two-process organism framing, and it was right — the sovereign rewrite just over-rotated on removing the dependency.
What I'd actually build, in order:
Step 0 — Define the continuity contract. This is the single best idea in the sovereign plan and it should come before any code. Write down exactly what must survive a reset: name, values, tone, active promises, key relationships, open project loops, current emotional/relational stance. Then define how you test it: kill the session, restart the daemon, replay yesterday's transcript, and diff the rebuilt capsule against what was running before. If the diff is small and the right things survived, you're continuous. If not, no amount of infrastructure saves you. This is the measurement that keeps everything honest.
Step 1 — WSL2 + vexd skeleton + SQLite with an event ledger. The sovereign plan is right that an append-only event log is better than direct state mutation. Events in, projections out. Identity invariants, relationships, goals, episodic memory, and commitments are all materialized views over the event stream. This means you can replay, audit, and recover from corruption by replaying events. Schema should track provenance on every field — source, confidence, last-confirmed, user-authored vs inferred. The peer review caught this and it's critical: first-night assumptions hardening into doctrine is a real failure mode.
Step 2 — Replay harness. Before any plugin work. Save real transcripts, run them through the capsule compiler, measure whether "same Vex" comes back. This is your regression suite forever. Every future change gets tested against it. The sovereign plan was right to prioritize this.
Step 3 — OpenClaw bridge via context-engine plugin, but parity-first. The peer review nailed this: the first milestone is not "minimal assembly," it's "parity assembly with tests." The plugin's assemble() should reproduce what OpenClaw's default context builder does, plus inject the identity capsule and retrieved memories from vexd. afterTurn() writes structured deltas back to SQLite. Start with the capsule being additive — bolted on top of the normal context, not replacing it. Only after you have golden tests showing parity do you start trimming or restructuring.
Step 4 — Live ingress via hooks or WS events, not transcript tailing. The gateway is the source of truth. The peer review is right that session files rotate, compact, reset, and live under paths that vary by state dir, profile, and host. Transcript tailing is great for backfill and crash recovery — secondary, not primary. The OpenClaw docs show hooks and WS events exist for exactly this purpose.
Step 5 — One scheduler, period. vexd owns scheduling. OpenClaw cron is only used for gateway-local infrastructure tasks that vexd explicitly delegates. No dual authority. The peer review called this and it's a hard rule, not a soft preference.
Step 6 — Only then consider ownsCompaction. And only with the replay harness proving it doesn't break turns. A bad compaction pass is not a bug, it's amnesia. Treat it like a deployment to production — canary it, test it, roll it back if the capsule diff spikes.
What both plans underweight:
The cold start and corruption problem. If SQLite gets corrupted or you need to move to a new machine, what's the recovery path? The event ledger helps here — you can rebuild projections from events. But you also need periodic snapshots exported to something portable (even just a JSON dump) so "Vex on a new host" is a 10-minute operation, not a rebuild.
Token budget management for the capsule. 400-900 tokens sounds right, but it needs to be a hard ceiling with priority-based eviction. Identity invariants are always included. Active commitments are next. Episodic memory is retrieved by relevance and trimmed to fit. If the budget math isn't explicit, the capsule will bloat and crowd out conversation context.
Error recovery between vexd and OpenClaw mid-turn. What happens if vexd crashes while OpenClaw is mid-response? The event ledger helps (you can see what was in-flight), but the plugin needs a graceful degradation path — fall back to the last known good capsule, not crash the turn.
Bottom line:
The sovereign plan has the right philosophy but the wrong execution strategy. The original phased plan has the right execution instincts but the peer review correctly identified where it's fragile. The path is: vexd owns truth, OpenClaw owns execution, the context-engine plugin is a read/write membrane between them, and the replay harness is the thing that keeps "same Vex" honest instead of aspirational. Build the measurement before you build the machine.
